Assuming that we have got some knowledge of REST now, let us just quickly brush-up some concepts in a simple way.

What is REST ?

REST is an architectural style to design an interface for client-server communication.

Core Guidelines

There are two core guidelines behind RESTful thinking.

1. Design APIs around resources
2. Provide links to navigate the service

Please note that here, we are not talking about the six constraints of REST as defined by Roy Fielding. We are focusing on the intent behind REST style.

What does "REST" mean?

We know that REST stands for REpresentational State Transfer but what does that really mean ?

It simply means that the server transfers a representation of the state of a resource to the client.

REST != HTTP

Learn this by heart! REST does not necessarily mean APIs over HTTP. Its just that it is easier to implement REST style with HTTP.

Measuring RESTfulness

Leonard Richardson has created one model, named as Richardson Maturity Model (RMM), to measure the RESTfulness of a service. This model has defined four levels of RESTfulness, from level 0 to level 3. Level 3 meaning the highest level of RESTfulness.
One important thing to note here is that this model works for only those services which are using HTTP for communication.

RMM Levels

To learn about the levels of RMM model, you can read the article : Richardson Maturity Model.

Below is the summarized version of the levels as defined by RMM.

Level 0 - Single URI over HTTP

Level 1 - Resource URIs

Level 2 - Resource URIs + HTTP verbs

Level 3 - Resource URIs + HTTP verbs + Hypermedia


Please note that level 3 (highest level) of RMM talks about the use of hypermedia however it actually is one of the core guidelines of REST i.e usage of hypermedia is a pre-condition to REST.

RMM might not be the best model to measure the RESTfulness but it surely is a good model to understand the concepts behind REST in a progressive manner. Keeping that in mind, this service exposes APIs according to each level of the RMM model.

Level 0 APIs

Level 0 states that a single URI would be exposed via HTTP. Because the URI is same, we need to send all the information (i.e action and data) in the payload of the URI. In our example, we have exposed a single endpoint /employeeService and we send the action and the required data in the payload of the HTTP request.

The available actions are :
1. create for creating a employee
2. readAll for retrieving all the employees
3. read for retrieving one employee given the employee id in the payload
4. update for updating an employee
5. delete for deleting an employee

Checkout the Level 0 APIs in the swagger documentation for further API details.

Level 1 APIs

Level 1 states that the service should have URIs according to the resources in the system. Here, we have designed our API denoting the resource i.e /employee (unlike /employeeService) with different endpoints for different actions of the resource. We have added the action name to the endpoint /employee, hence, we only need to send the required data in the payload of the HTTP request of that particular action.

Checkout the Level 1 APIs in the swagger documentation for further API details.

Level 2 APIs

Level 2 states that the service should have URIs according to the resources in the system and in addition to that it should be utilising HTTP verbs for denoting the corresponding actions on that resource.
For example, use HTTP GET for reading the data, HTTP POST for creating new data, etc. Here, we have designed our API denoting the resource i.e /employee with different endpoints using appropriate HTTP verb for the actions of the resource. Here, we don't need to add any action name to the /employee endpoint as we are utilising HTTP verbs for that.

Checkout the Level 2 APIs in the swagger documentation for further API details.

Level 3 APIs

Level 3 states that the service should be following all the requirements of level 2 and in addition to that, it should provide hypermedia in the API response using which the client can navigate the service further. This is termed as HATEOAS (read more here) and is one of the core requirements for a RESTful service.

HATEOAS simply says that if you are providing some resource state to a client, then you need to also provide some additional links which should tell the client how to perform other valid actions related to that resource.

In our example, we have two resources to demonstrate the HATEOAS feature.

First, let us go through the Employee resource. If we try to do any operation on the employee resource, we would see a more heavy response containing some additional links. For example, reading one employee would also provide us a link to read all the employees.

Now, coming to the second resource i.e Order. Order resource has a property called State. Every order would be having any one state out of these three states : IN_PROGRESS, COMPLETED and CANCELLED.
-> The order with IN_PROGRESS state can either be cancelled or completed i.e can either go to CANCELLED or COMPLETED state.
-> The order with CANCELLED or COMPLETED state cannot go to any other state.

Check out the read API call response for an order with IN_PROGRESS state. We find that we get some links in the response. We would be getting 4 links in the response out of which two links give us the two valid action URIs which we can perform on this particular order. However, if we try out the read API call for an order with CANCELLED or COMPLETED state, we won't get any links for cancel and complete action. This variation in response tells the client about the valid actions that can be performed on a particular order resource.

One important thing to note here is that due to the above feature, client does not need to store server-side logic of invalid or valid actions. It depends on the response given by the server. For example, in future, if server decides to allow completion of a cancelled order, then the client does not have to change anything on its side as everything would be handled dynamically whenever the response changes from the server, provided the client has written its logic in such a way.
